# 用不同测试类型 Prompt，把一场“京东线上活动秒杀礼品”测明白

我一直觉得，很多 QA Prompt 写得很努力，但不太能直接拿来干活——内容很全，落地很慢，最后还是得人手改一遍。这个仓库里不同测试类型的 Prompt，价值不在“它会写很多字”，而在“它把不同测试任务拆开了”。这事很重要——需求分析不是测试用例，测试策略不是缺陷报告，性能测试也不该和功能测试混成一锅。

这篇文章就做一件事：把这些测试类型 Prompt 讲清楚——它们分别解决什么问题，适合什么场景，用到什么程度，以及放到一个真实业务里到底会吐出什么东西。

我选的模拟业务场景很简单，也很麻烦——京东线上活动秒杀礼品。简单，是因为业务很好理解。麻烦，是因为它天然带着高并发、库存一致性、资格校验、风控、支付链路、履约状态、活动配置、前后端一致性这些坑。也就是说，这个场景非常适合拿来验 Prompt 到底是不是纸上谈兵。

---

## 先说结论

如果你只想知道怎么选：

| 测试类型 | 解决的问题 | 最适合什么时候用 |
|---|---|---|
| 需求分析 | 先把业务、范围、风险和测试场景盘清 | 需求评审刚结束 |
| 测试策略 | 决定测什么、不测什么、怎么排优先级 | 项目启动或活动排期前 |
| 测试用例编写 | 把场景变成可执行用例 | 进入测试设计阶段 |
| 功能测试 | 把功能正确性和主流程覆盖做扎实 | 提测后主线验证 |
| API 测试 | 把资格、库存、下单、支付等接口测透 | 服务接口稳定后 |
| 自动化测试 | 把核心回归变成机器活 | 版本进入持续迭代 |
| 性能测试 | 提前看秒杀高峰能不能扛住 | 大促、活动上线前 |
| 安全测试 | 看资格绕过、刷接口、越权和数据泄露 | 活动开放前 |
| 手动测试 | 补自动化和脚本测不到的细节 | 联调、体验验证、上线前走查 |
| 移动端测试 | 专门处理 App、H5、设备和网络问题 | 业务在移动端承载时 |
| 可访问性测试 | 确保关键流程不把特殊用户挡在门外 | 活动面向大众用户时 |
| 缺陷上报 | 把问题写到开发能立刻改 | 发现缺陷后 |
| 测试报告 | 给项目和业务方一个可决策结果 | 阶段收口、上线前 |
| 测试用例评审 | 看现有用例到底漏了多少坑 | 用例写完但还没执行前 |
| AI 辅助测试 | 用 AI 帮你挑重点、补场景、提效率 | 资料多、节奏快、资源紧时 |

换句话说——这不是 15 个长得差不多的 Prompt。这是 15 个不同测试动作的入口。

## 这个模拟业务，先定一下边界

业务名称：京东线上活动秒杀礼品  
业务目标：在指定活动时段，符合资格的用户可以 0 元或低价秒杀指定礼品  
核心链路：活动页曝光 → 用户登录 → 资格校验 → 库存锁定 → 下单 → 支付/确认 → 履约发货  
主要风险：

- 活动开始瞬间流量打爆
- 库存超卖或少卖
- 用户资格绕过
- 下单成功但支付/履约状态异常
- 前端展示和服务端真实库存不一致
- 活动配置错误导致礼品、价格、时间、地域限制失真

这几个点，基本就是整个博客后面所有 Prompt 的共同输入。

## 1. 需求分析 Prompt——先别急着写用例，先把坑挖出来

这个 Prompt 最值钱的地方，是它会逼你把“活动规则”说清楚。很多秒杀问题，不是代码错了，是需求本身就没讲透。

### 适用场景和范围

- 活动 PRD 刚出来
- 活动配置规则复杂
- 多角色参与——产品、运营、研发、测试都在场
- 需要先产出测试场景池，而不是直接开测

### 这类 Prompt 通常会产出什么

- 业务背景
- 测试范围
- 正向 / 异常 / 边界 / 风险场景
- 测试重点和依赖

### 模拟效果

输入给 Prompt 的核心业务：

```markdown
活动名称：618 秒杀礼品活动
礼品：京东定制保温杯、充电宝、PLUS 联名周边
活动规则：
- 每天 10:00 / 16:00 / 20:00 三场
- 每个用户每天限抢 1 次
- 仅 PLUS 用户和活动邀请用户可参与
- 库存有限，先到先得
- 下单后 15 分钟内未支付则释放库存
```

模拟输出片段：

```markdown
关键测试场景：
1. PLUS 用户在活动开始后成功秒杀并完成支付
2. 非 PLUS 用户点击抢购按钮时被拦截
3. 邀请用户资格在活动页展示为可参与，但下单接口返回无资格
4. 用户在库存即将耗尽时并发下单，验证是否发生超卖
5. 用户下单后超时未支付，库存是否正确回补
6. 用户跨场次、跨天参与时，限抢规则是否按预期生效
7. 活动配置时间与客户端倒计时不一致时，是否出现提前抢购
```

这一步的价值很实际——它不是帮你写漂亮话，它是帮你避免后面整轮测试都在错方向上忙。

## 2. 测试策略 Prompt——决定资源砸在哪，不然就会平均用力

秒杀活动最怕平均主义。什么都测一点，最后真正危险的高并发链路没测透。

### 适用场景和范围

- 活动测试周期很短
- 人手有限
- 需要明确 P0 / P1 / P2
- 需要对业务方解释为什么某些内容只做抽测

### 模拟效果

```markdown
测试优先级建议：
- P0：资格校验、库存扣减、下单支付链路、库存回补
- P1：活动页展示、倒计时、弹窗提示、订单状态同步
- P2：礼品详情页、非核心推荐位、活动分享链路

建议不纳入本轮深测范围：
- 非活动商品常规购买链路
- 历史活动页静态内容
- 非本次活动礼品的履约规则
```

这就是测试策略的意义——它不给你“更努力”，它给你“更集中”。

---

## 3. 测试用例编写 Prompt——把想法落成能执行的东西

有了场景，不代表能执行。测试用例 Prompt 的作用，就是把抽象风险压成具体步骤、数据和预期结果。

### 适用场景和范围

- 已经有需求和场景
- 需要交付标准化测试资产
- 需要多人协作执行

### 模拟效果

```markdown
用例标题：PLUS 用户在活动开始后成功秒杀礼品
前置条件：
- 用户账号为 PLUS
- 当前场次礼品库存 > 0
- 用户当日未参与过秒杀

步骤：
1. 进入活动页并等待 10:00 开场
2. 点击“立即秒杀”
3. 提交订单并完成支付

预期结果：
- 抢购按钮在开场后可点击
- 下单成功，订单状态为“待支付/已支付”
- 库存数量减少 1
- 用户再次进入活动页时提示“今日已参与”
```

这种输出很朴素——但朴素是好事。用例不是文学创作。

## 4. 功能测试 Prompt——主线通不通，文案、状态和规则是不是一回事

功能测试 Prompt 对这种活动特别有用，因为活动业务表面简单，状态其实很多。

### 适用场景和范围

- 活动页、商品页、订单页、支付页联动
- 前后端状态多
- 规则变化频繁

### 模拟效果

```markdown
功能测试重点：
- 活动未开始、进行中、已结束三种状态切换
- 库存充足、库存紧张、库存售罄三种展示
- 用户资格命中、未命中、已参与三类状态分流
- 下单成功、支付超时、支付失败、订单取消四类结果闭环
```

如果需求分析是挖坑，功能测试就是拿铲子下去确认坑到底有多深。

---

## 5. API 测试 Prompt——很多秒杀事故，最后都死在接口

活动页漂不漂亮，其实不是最要命的。资格接口、库存接口、下单接口、支付回调接口，这几条链路一旦出问题，线上就是事故。

### 适用场景和范围

- 微服务架构
- 活动资格和库存规则在服务端
- 需要验证接口契约、幂等、错误码、状态一致性

### 模拟效果

```markdown
核心接口清单：
- GET /activity/seckill/config
- POST /activity/seckill/qualification/check
- POST /activity/seckill/stock/lock
- POST /activity/seckill/order/create
- POST /payment/callback

关键验证点：
- 重复点击“立即秒杀”是否重复锁库存
- 下单接口在无资格情况下是否返回明确错误码
- 支付超时后库存回补是否幂等
- 回调重复投递是否导致订单状态抖动
```

这类 Prompt 的好处，是它能把“看起来是一个按钮”的事情拆成 5 到 8 个服务动作。你一旦这么看，问题会清楚很多。

## 6. 自动化测试 Prompt——把最贵的回归变成便宜的回归

秒杀活动不是只上一天。大多数团队会反复改活动时间、资格规则、礼品配置、前端样式。手工每轮都回归，成本很快炸掉。

### 适用场景和范围

- 活动反复迭代
- 核心回归链路稳定
- 希望把开场前 smoke 变成自动执行

### 模拟效果

```markdown
自动化优先级建议：
- P0 自动化：资格校验、下单主链路、支付超时回补、库存售罄提示
- P1 自动化：活动状态切换、订单状态同步、活动配置生效校验
- P2 手动保留：视觉样式、复杂弱网交互、运营临时配置检查
```

这个思路是对的——自动化不该贪多。先把最贵、最常回归、最容易翻车的部分机器化。

---

## 7. 性能测试 Prompt——这类活动不上压测，基本就是赌

秒杀业务天然带峰值流量，问题不在平均值，在那几秒钟。

### 适用场景和范围

- 大促、整点抢购、限量礼品活动
- 活动链路需要并发验证
- 需要判断扩容是否足够

### 模拟效果

```markdown
性能场景建议：
- 09:59:50 到 10:00:10 的流量突刺压测
- 高并发下资格校验和库存锁定接口联合压测
- 支付超时批量释放库存的补偿任务压测

监控重点：
- 接口 RT/P95/P99
- 库存锁失败率
- 下单成功率
- 消息堆积
- Redis / DB 热点键
```

如果一个性能 Prompt 连“活动开始前后 20 秒”都没强调，那它基本没抓到这个业务最危险的地方。

---

## 8. 安全测试 Prompt——秒杀活动永远有人想绕规则

资格绕过、参数篡改、重放请求、批量刷接口、伪造活动身份，这些都不是理论风险——这就是线上会发生的事。

### 适用场景和范围

- 有资格门槛
- 有库存稀缺资源
- 有账户体系和支付链路

### 模拟效果

```markdown
安全测试重点：
- 修改请求参数中的 userLevel，验证是否能伪装 PLUS 资格
- 重放抢购请求，验证是否可重复占库存
- 越权查询他人订单状态
- 高频调用资格接口，验证风控和限流是否生效
```

说白了——你在设计活动，别人也在设计绕过活动。

---

## 9. 手动测试 Prompt——自动化不太会觉得“这个流程看着就别扭”

活动页尤其吃体验。按钮文案、倒计时节奏、资格拦截提示、抢空后的兜底文案，这些东西不一定是 bug，但很可能会出投诉。

### 适用场景和范围

- 活动页体验走查
- 多端对比
- 弱网、异常操作、真实手速验证

### 模拟效果

```markdown
探索点：
- 倒计时结束前 1 秒反复点击，按钮状态是否抖动
- 售罄后返回活动页，文案是否仍显示“立即秒杀”
- 用户资格不足时，提示是否告诉用户为什么不能参与
- App、H5、小程序之间展示和规则说明是否一致
```

很多线上差评，严格说不是功能缺陷——是产品和实现一起把用户搞烦了。

---

## 10. 移动端测试 Prompt——别把 App 和 H5 当成同一个东西

这个业务大概率主要跑在移动端。那你不测机型、系统版本、弱网、前后台切换、通知回跳，基本就是在装没事。

### 模拟效果

```markdown
移动端重点：
- iPhone 低版本 Safari 与 Android Chrome 的活动页兼容
- 弱网下点击秒杀按钮后的重复提交风险
- App 从后台回到前台时倒计时是否刷新
- 小程序内支付返回后订单状态是否同步刷新
```

---

## 11. 可访问性测试 Prompt——活动面向所有用户，不是只面向手速快的人

这个模块很多团队会跳过，但活动页如果连焦点、朗读顺序、按钮可识别性都做不好，本质上就是把一部分用户排除在外。

### 模拟效果

```markdown
无障碍重点：
- 秒杀按钮是否有明确可访问名称
- 倒计时信息是否能被屏幕阅读器正确朗读
- 售罄状态是否只靠颜色区分
- 资格失败提示是否能被辅助技术读取
```

## 12. 缺陷上报 / 测试报告 / 用例评审 / AI 辅助测试——这些不是“补充模块”，它们是收口模块

这四类 Prompt 往往被低估，但它们决定团队最后到底是不是在同一张纸上工作。

### 缺陷上报 Prompt

适合把“库存已售罄但仍可创建订单”这种问题写成开发能立刻定位的报告。

模拟输出：

```markdown
标题：秒杀库存售罄后仍可创建订单
严重级别：Critical
前置条件：活动场次库存为 0
实际结果：客户端提示售罄，但接口返回下单成功
影响：存在超卖风险
```

### 测试报告 Prompt

适合上线前给业务和项目经理一个“能不能发”的版本判断。

模拟输出：

```markdown
测试结论：
- P0 问题已关闭
- 高并发场景下库存锁定成功率符合预期
- 仍存在 2 个 P1 体验问题，不阻塞上线
上线建议：可上线，需保留库存与支付链路重点监控
```

### 测试用例评审 Prompt

适合在执行前发现“你写了 30 条用例，但没人测支付超时回补”这种大漏项。

模拟输出：

```markdown
评审结论：
- 主链路覆盖较完整
- 缺失场景：跨场次限抢、支付回调重复、弱网重复点击、资格缓存失效
- 风险等级：高
```

### AI 辅助测试 Prompt

适合在活动需求和变更很多时，先帮你找高风险点和回归重点。

模拟输出：

```markdown
AI 建议优先回归模块：
1. 活动资格规则
2. 库存锁定与释放
3. 订单状态同步
4. 活动倒计时与前端按钮状态
```

## 最后一句——Prompt 不是替你测试，它是替你把测试拆对

我对 QA Prompt 的标准其实不高。它不用显得聪明，也不用讲得像咨询公司报告。它只要做到一件事——让测试团队更快进入正确的问题空间。

这个仓库里不同测试类型的 Prompt，真正有用的地方就在这。你不是把一个万能 Prompt 喂给 AI，然后期待它懂你所有上下文。你是把一个复杂业务，拆成需求分析、策略、用例、接口、性能、安全、报告这些明确动作，再让 AI 在每一个动作里帮你省时间、补盲区、压结构。

这才是 Prompt 在测试里该干的活——不是替代判断，是放大判断。

如果你真要开始用，我的建议也很直接：

- 先从 `Requirements Analysis` 开始——别一上来写用例
- 默认选 `Standard-version/`
- 活动类业务一定把 `API / Performance / Security / Manual` 四个模块一起用
- 上线前一定补 `Test Reporting`

秒杀活动这种业务，真正难的从来不是“怎么写测试文档”。真正难的是——在很短时间里，把真正会出事故的地方抓住。

Prompt 写得对，这事会轻松很多。
